**Branch**: `006-menu-check-state-indicator` | **Date**: 2026-06-19 | **Spec**: [spec.md](spec.md)

**Input**: Feature specification from `specs/006-menu-check-state-indicator/spec.md`

## Summary

Add a `✓` prefix to toggleable View menu items when their associated toggle is active. The immediate
trigger is the deferred FR-001 from feature 005: "Soft Wrap (ext)" in the View menu shows no
visual state when soft-wrap is on. The implementation adds `toggle_states: &'a [(Action, bool)]`
to `MenuBarWidget`, passes the app's current toggle booleans at render time, and widens the
dropdown by 2 chars when the menu is checkable-aware. No new crate dependencies; two files changed,
one test module added.

## Implementation Phases

### Phase A: `MenuBarWidget` — add `toggle_states` field and update constructor

**File**: `src/ui/menubar.rs`

**Tasks**:
1. Add `pub toggle_states: &'a [(Action, bool)]` field to `MenuBarWidget<'a>` struct.
2. Update `MenuBarWidget::new()` signature to accept `toggle_states: &'a [(Action, bool)]`.
3. Assign `toggle_states` in the constructor body.

**Why no changes to `MenuItem`**: `MenuItem` items are `static` Rust constants. Runtime state
cannot be stored in them. The `toggle_states` slice on the widget is the correct pattern (same
as how `EditorWidget` receives `wrap_starts` — a runtime-computed slice not stored in the
static editor config).

---

### Phase B: Dropdown render logic — prefix column

**File**: `src/ui/menubar.rs` — `Widget::render()` impl, `DropDown` branch (lines ~401–486)

**Tasks**:
1. Add helper: `fn lookup_checked(toggle_states: &[(Action, bool)], action: &Action) -> Option<bool>` —
   returns `Some(true)`, `Some(false)`, or `None` (action not in slice).
2. Compute `has_checkable: bool` — `true` if any item in the current `menu_items` has an
   action that returns `Some(_)` from `lookup_checked`.
3. Adjust `content_width`:
   - `has_checkable = false`: unchanged (`max_label_len + 4`)
   - `has_checkable = true`: `max_label_len + 6`
4. In the per-item render loop, before writing the label:
   - If `has_checkable`:
     - Determine prefix: `lookup_checked(toggle_states, &item.action)` → `Some(true)` →
       write `'✓'` at `label_x`, `' '` at `label_x + 1`; otherwise write `' '`, `' '`.
     - Set `label_x` to `area.left() + start_col + 3` (was `+ 1`).
   - If `!has_checkable`: `label_x` unchanged at `area.left() + start_col + 1`.

**Key invariant**: `has_checkable` is computed once per dropdown open, not per item. All items in
the same dropdown share the same `label_x` column (FR-008 alignment).

### Phase D: Unit tests

**File**: `src/ui/menubar.rs` — new `#[cfg(test)]` module at bottom of file

**Tests** (7 total, per `contracts/menu-widget.md` + tasks.md):

1. **`test_checkmark_shown_when_toggle_true`** — View dropdown, `soft_wrap = true`; assert prefix cell `'✓'`.
2. **`test_no_checkmark_when_toggle_false`** — View dropdown, `soft_wrap = false`; assert prefix cell `' '`.
3. **`test_non_toggle_menu_unaffected`** — File dropdown; assert no `'✓'` and no width expansion (width = `max_len + 4 = 23`).
4. **`test_label_alignment_in_checkable_menu`** — View dropdown; assert all labels start at same column; assert absent-from-toggle-states items get `' '` prefix (FR-003 absent case).
5. **`test_empty_toggle_states_no_regression`** — View dropdown, `toggle_states = &[]`; assert no `✓`, width unchanged.
6. **`test_second_action_also_shows_checkmark`** — Options dropdown, `toggle_states = &[(Action::ToggleHighlight, true)]`; assert `'✓'` prefix (FR-007 generality proof).
7. **`test_initial_soft_wrap_state_from_config`** — `toggle_states = &[(Action::ToggleSoftWrap, true)]` without prior toggle; assert `'✓'` (US3 config-init path).
